# RPA 页面适配演示用例

## 用例 1：CSS 选择器失效后适配成功

准备：

- Forms 节点值设置为 `{{variables@Sheet1.20.18}}`。
- Forms 节点 CSS 选择器故意设置为一个不存在的选择器。
- 开启“CSS匹配失败后启用税务页面适配”，阈值设置为 `0.75`。
- 页面中保留包含“第11栏”“销项税额”“一般项目本月数”的空输入框。

预期：

- 原 CSS 选择器匹配失败。
- 系统根据 `Sheet1.20.18` 识别为主表第 11 栏“销项税额”。
- 页面适配命中同一行的空输入框并完成填报。
- 日志记录 `forms-adaptive-match`，包含业务字段、置信度、匹配依据。

## 用例 2：页面控件已有值时不覆盖

准备：

- 使用与用例 1 相同的 Forms 节点配置。
- 页面中所有可编辑输入框都预先填入值。

预期：

- 原 CSS 选择器匹配失败。
- 页面适配扫描不到未填写的可编辑控件。
- 系统不覆盖任何页面值，**该 Forms 节点跳过填报但工作流继续向下执行**。
- 日志记录 `forms-adaptive-match`，`reason: no-empty-controls`，并附带原选择器、Excel 坐标和业务字段说明。

## 用例 3：置信度不足时停止填报

准备：

- Forms 节点值设置为 `{{variables@Sheet2.20.18}}`。
- 页面存在空输入框，但附近没有附列资料一、行文本、列文本或字段语义。

预期：

- 系统能识别 Excel 坐标和 Sheet 上下文。
- DOM 候选控件最高置信度低于阈值。
- 系统不执行填报，**该 Forms 节点跳过填报但工作流继续向下执行**。
- 日志记录 `forms-adaptive-match`，`reason: confidence-too-low`，并附带候选控件的评分明细。

## 用例 4：多个候选接近时停止填报

准备：

- 页面中存在两个空输入框，附近文本都包含相同的行号或字段语义。
- 两个候选控件的评分差距小于 `0.08`。

预期：

- 系统返回 `reason: ambiguous-match`。
- **该 Forms 节点跳过填报但工作流继续向下执行**，提示用户修正 CSS 选择器或优化页面匹配条件。
- 日志记录候选控件评分明细，便于诊断歧义来源。

> 备注：以上 2、3、4、5 用例中，"跳过填报但继续执行" 的设计目的是让用户在日志中看到完整的适配计算结果，并由后续节点或人工补救；只有完全没有进入适配（例如 CSS 失败但开关未开启）时，才会以 `element-not-found` 终止工作流。

## 用例 5：附表动态上下文匹配

准备：

- Forms 节点值设置为 `{{variables@Sheet2.20.18}}`。
- `variables.Sheet2` 中第 20 行和第 18 列附近存在“销售额”“销项税额”“合计”等表格上下文。
- 页面中存在对应语义的空输入框。

预期：

- 系统无法从静态 schema 找到附表明细字段时，使用 Sheet 矩阵上下文生成关键词。
- 页面适配根据行文本、列文本、控件上下文计算置信度。
- 置信度达标后执行填报，并在日志中说明该字段来源为 `sheet-context`。
